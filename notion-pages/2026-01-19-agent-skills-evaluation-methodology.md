## 1. 평가의 필요성과 도전 과제

### 왜 에이전트 평가가 중요한가?

- **시장 성장**: Agentic AI 시장은 2026년까지 100억 달러 이상, 2034년까지 약 2,000억 달러로 성장 예상
- **높은 실패율**: 2024-2025년 AI 프로젝트의 39%가 기대에 미치지 못함
- **도입 가속화**: 60% 이상의 조직이 현재 AI 에이전트를 도입 중

### 에이전트 평가의 도전 과제

**기존 소프트웨어와의 차이점:**

| 기존 소프트웨어 | AI 에이전트 |
|----------------|------------|
| 결정적(Deterministic) | 확률적(Probabilistic) |
| 고정된 입출력 | 다양한 경로로 동일 결과 가능 |
| 단일 실행 테스트 충분 | 다회 실행 평가 필요 |
| 명확한 정답 존재 | 주관적 품질 평가 필요 |

**주요 도전 과제:**

- 비결정적 출력으로 인한 재현성 문제
- 다단계 작업의 중간 과정 평가
- 도구 사용의 정확성과 효율성 측정
- 안전성과 신뢰성의 정량화

## 2. 평가 프레임워크

### 2.1 2차원 분류법 (KDD 2025)

KDD 2025 튜토리얼에서 제안된 체계적 프레임워크:

**축 1: 평가 목표 (What to Evaluate)**
- 에이전트 행동 (Behavior)
- 능력 (Capabilities)
- 신뢰성 (Reliability)
- 안전성 (Safety)

**축 2: 평가 과정 (How to Evaluate)**
- 상호작용 모드
- 데이터셋 및 벤치마크
- 메트릭 계산 방법
- 도구 및 인프라

### 2.2 CLASSic 프레임워크 (ICLR 2025)

엔터프라이즈 AI 에이전트 평가를 위한 5차원 프레임워크:

| 차원 | 설명 |
|------|------|
| **C**ost | API 사용량, 토큰 소비, 인프라 비용 |
| **L**atency | 엔드투엔드 응답 시간 |
| **A**ccuracy | 워크플로우 선택 및 실행 정확도 |
| **S**tability | 다양한 입력에 대한 일관성 및 견고성 |
| **S**ecurity | 적대적 입력, 프롬프트 주입, 데이터 유출 저항성 |

### 2.3 계층별 평가 모델

AI 에이전트의 두 계층을 분리하여 평가:

**추론 계층 (Reasoning Layer):**
- LLM이 담당하는 계획 및 의사결정
- 평가 질문: 효과적인 계획을 세우는가? 계획의 범위가 적절한가?

**행동 계층 (Action Layer):**
- 도구(함수 호출)를 통한 실제 실행
- 평가 질문: 올바른 도구를 선택했는가? 올바른 파라미터를 전달했는가?

## 3. 주요 벤치마크

### 3.1 범용 에이전트 벤치마크

#### AgentBench
- **설명**: LLM을 에이전트로 평가하는 최초의 다차원 벤치마크
- **환경**: 8개의 독립된 환경에서 추론 및 의사결정 능력 평가
- **평가 대상**: 29개 API 기반 및 오픈소스 LLM
- **GitHub**: https://github.com/THUDM/AgentBench

#### GAIA (General AI Assistant)
- **설명**: 범용 AI 어시스턴트 벤치마크
- **특징**: 단계별 계획이 필요한 복잡한 실제 질의 시뮬레이션
- **평가 영역**: 추론, 검색, 작업 실행의 조합 능력

### 3.2 웹 에이전트 벤치마크

#### WebArena
- **설명**: 자율 웹 에이전트 평가를 위한 포괄적 스위트
- **규모**: 812개 장기 작업 (241개 템플릿 기반)
- **환경**: 전자상거래, 소셜 포럼 등 다양한 웹 환경 복제
- **발전 현황**: 2년 만에 성공률 14% → 약 60%로 향상 (2025년 2월 기준 61.7%)

#### MiniWob++
- **설명**: 합성 웹 자동화 작업 벤치마크
- **용도**: 기본적인 웹 상호작용 능력 테스트
- **특징**: BrowserGym 프레임워크와 호환

#### WorkArena
- **설명**: 엔터프라이즈 웹 워크플로우 시뮬레이션
- **규모**: 682개 작업
- **특징**: ServiceNow 포털을 통한 노트북 주문 등 실제 업무 시나리오
- **발견점**: MiniWob++에서 잘 수행하는 에이전트도 복잡한 다단계 작업에서는 실패

### 3.3 소프트웨어 엔지니어링 벤치마크

#### SWE-bench
- **설명**: GitHub의 실제 소프트웨어 이슈 해결 능력 평가
- **규모**: 12개 Python 저장소에서 2,294개 이슈-커밋 쌍
- **변형**: SWE-bench Lite (300개 인스턴스로 축소된 버전)
- **입력**: 코드베이스 + 이슈 설명 → 패치 생성

#### MLGym
- **설명**: AI 연구 에이전트 평가를 위한 최초의 Gym 환경
- **특징**: 다양한 개방형 AI 연구 작업 (MLGym-Bench)
- **메트릭**: 다양한 작업에서 여러 에이전트 비교를 위한 새로운 평가 지표

### 3.4 기타 특화 벤치마크

| **TheAgentCompany** | 연속적인 다단계 작업, 부분 점수 부여 |

## 4. 평가 메트릭

### 4.1 작업 완료 메트릭

**Success Rate (SR) / Task Success Rate (TSR):**
- 에이전트가 사용자 목표를 성공적으로 달성했는지 측정
- 가장 일반적으로 사용되는 메트릭
- 한계: 이진 지표로 부분적 진행 상황 미반영

**Pass@k:**
- k번의 시도 중 최소 1회 성공 여부
- 한계: 모든 실패를 동등하게 취급

**Goal Completion Rate:**
- 다단계 작업의 엔드투엔드 완료율

### 4.2 궤적 품질 메트릭

**Agent Trajectory:**
- 에이전트가 합리적이고 효과적인 경로를 따르는지 평가
- "Silent Failure" 감지: 올바른 출력이지만 비효율적/잘못된 과정

**중간 체크포인트:**
- 워크플로우의 주요 노드에서 성공률 측정
- 부분 점수 부여로 세밀한 평가 가능

### 4.3 도구 사용 메트릭

**Tool Selection Accuracy:**
- 올바른 도구를 선택했는가?

**Tool Use Efficiency (TUE):**
- API, 데이터베이스, 계산기 등을 얼마나 효율적으로 사용하는가?

**Parameter Accuracy:**
- 올바른 정보를 올바른 컨텍스트에서 전달했는가?

### 4.4 추가 평가 차원

| 메트릭 | 설명 |
|--------|------|
| **Memory & Recall** | 이전 컨텍스트나 단계를 기억하는가? |
| **Adaptability** | 예상치 못한 입력이나 엣지 케이스에서 복구 능력 |
| **Latency vs Quality** | 실시간 속도 제약 하에서 최적화 능력 |
| **TCRR** | 중복 제거율 |
| **GSRT** | 목표 변경 복구 시간 |
| **Human Feedback Alignment** | 인간이 에이전트의 솔루션 흐름을 선호하는가? |

### 4.5 비용 효율성 메트릭

- 토큰 소비량
- API 호출 비용
- 인프라 오버헤드
- Aider Polyglot 벤치마크: 현재 점수 대비 비용을 비교하는 유일한 벤치마크

## 5. 평가 도구 및 플랫폼

### 5.1 주요 평가 플랫폼 비교

| 플랫폼 | 최적 용도 | 특징 |
|--------|----------|------|
| **LangSmith** | LangChain/LangGraph 팀 | 네이티브 워크플로우 트레이싱, 최소 설정 |

| **Langfuse** | 오픈소스 필요 팀 | MIT 라이선스, 셀프 호스팅 가능 |

### 5.2 LangSmith

**장점:**
- LangChain 생태계 완벽 통합
- 환경변수 하나로 자동 트레이싱
- 복잡한 다단계 워크플로우 시각화

**주요 기능:**
- 검색된 문서, 컨텍스트 조합, 도구 호출, 최종 출력 추적
- 빠른 프로토타이핑에 적합

### 5.3 Braintrust

**사용 기업:** Notion, Stripe, Vercel, Airtable, Instacart, Zapier, Coda 등

**주요 기능:**
- GitHub Action을 통한 네이티브 CI/CD 통합
- PR에 실험 결과 자동 게시
- 프롬프트 변경의 나란히 비교

### 5.4 Arize

**제품군:** Arize Phoenix, Arize AX

**강점:**
- 전통적 ML 모니터링 경험 (모델 드리프트, 바이어스 모니터링)
- 에이전트 평가에 필요한 트레이스 및 세션 온라인 평가
- 오픈소스 평가 라이브러리

### 5.5 DeepEval

**특징:**
- AI 에이전트 평가에 특화된 프레임워크
- 사전 정의된 기준과 메트릭으로 자동화된 평가
- LLM-as-a-Judge 기반 스코어링

## 6. LLM-as-a-Judge 방법론

### 6.1 개요

LLM-as-a-Judge는 정의된 기준에 따라 생성된 텍스트를 평가하기 위해 LLM을 사용하는 평가 방법입니다. 챗봇, Q&A 시스템, 에이전트 등 모든 LLM 기반 제품의 출력 품질을 평가할 수 있습니다.

### 6.2 Judge 유형

**단일 출력 스코어링 (참조 없음):**
- Judge LLM이 입력만으로 루브릭 기반 점수 부여
- 용도: 주관적 품질 평가

**단일 출력 스코어링 (참조 있음):**
- 골드 스탠다드 "예상 출력"과 비교
- 장점: 일관성 향상

**쌍별 비교 (Pairwise):**
- 동일 입력에 대한 두 출력 비교
- 용도: A/B 테스트, 모델 비교

### 6.3 평가 기준 예시

일반적으로 평가하는 품질:
- 유창성 (Fluency)
- 정확성 (Correctness)
- 관련성 (Relevance)
- 사실성 (Factuality)
- 지시 준수 (Instruction Following)
- 톤, 명확성, 형식 준수, 간결성, 정중함
- PII 포함 여부

### 6.4 루브릭 설계 베스트 프랙티스

**G-Eval 접근법 (Liu et al., EMNLP 2023):**
1. 작업 소개와 평가 기준만 제공
2. LLM이 상세한 평가 단계의 Chain-of-Thought 생성
3. 각 점수 범주에 대한 명확한 설명 포함

**효과적인 루브릭 요소:**
- 범주형 정수 스코어링 스케일 사용
- 각 점수의 의미에 대한 명확한 설명
- Few-shot 예시로 채점 메커니즘 보정
- 가능한 각 점수의 logprobs 측정하여 가중 출력 계산

### 6.5 프롬프트 구조

**역할 부여:**
```
당신은 기술 지원 품질 보증 분야에서 10년 경력의 시니어 고객 경험 전문가입니다.
```

**구체적 기준 제공:**
```
다음 기준으로 응답을 평가하세요:
- 사과하지 않고 명확하고 간결한 답변을 제공하는가
- 기술적으로 정확한가
- 사용자의 질문에 직접적으로 답변하는가
```

### 6.6 한계 및 고려사항

**자기 선호 편향:**
- GPT-4, Claude-v1 등은 자신의 응답을 10-25% 더 선호
- 해결책: 교차 평가, 다중 Judge 사용

**재현성:**
- 공통 루브릭, 프롬프트 형식, 메타데이터 로깅 필요
- Cohen's Kappa, Krippendorff's Alpha 등 평가자 간 신뢰도 측정

### 6.7 Agent-as-a-Judge 진화

다중 에이전트 평가 프레임워크:
- 여러 LLM 에이전트가 협업하거나 토론하여 출력 평가
- 도메인 전문가, 비평가, 옹호자 등 다양한 역할 수행
- 인간 심사위원 패널과 유사한 다각적 평가

## 7. 오프라인 vs 온라인 평가

### 7.1 오프라인 평가

**특징:**
- 기존 소프트웨어의 유닛/회귀 테스트와 유사
- 사전 라벨링된 고정 데이터셋 사용
- API 호출은 보통 모킹 또는 페이킹

**적합한 경우:**
- "정답"을 명확히 정의할 수 있을 때
- 분류, 감지, 추출, 랭킹 작업
- 회귀 테스트: 새 버전이 성능을 개선/저하시키는지 확인

**한계:**
- 현실성 부족: 엣지 케이스, 새로운 사용자 행동 미반영
- 프로덕션과 다를 수 있음

### 7.2 온라인 평가

**특징:**
- 실제 프로덕션 환경에서 평가
- A/B 테스트, 단계적 롤아웃, 모니터링 기반 비교

**장점:**
- 실제 지연 시간, 분포 변화, 사용자 피드백 루프 포착
- 오프라인 데이터셋에 나타나지 않는 문제 발견

**권장 사용:**
- 오프라인 신뢰도 확보 후 프로덕션 검증

### 7.3 A/B 테스트 전략

**단계별 접근:**

1. **카나리 트래픽 및 섀도우 테스팅**: 사용자 노출 없이 시작
2. **소량 트래픽 테스트**: 오프라인 게이트 통과 후 5-10% 트래픽
3. **롤백 트리거**: 지연/비용 회귀 또는 안전성 위반 시 자동 롤백

**베스트 프랙티스:**
- 사전 릴리스 시뮬레이션으로 약한 변형 필터링
- AI 게이트웨이를 통한 자동 폴백으로 카나리 트래픽 라우팅
- 실시간 알림과 함께 롤백 임계값 적용

## 8. 실전 베스트 프랙티스

### 8.1 메트릭 선정

**권장 조합 (3-5개 메트릭):**
- 컴포넌트 레벨: 도구 정확도, 파라미터 정확도
- 엔드투엔드: 작업 완료율
- 최소 1개의 커스텀 메트릭 개발 (G-Eval 등)

### 8.2 트레이싱 및 관찰성

**필수 로깅 항목:**
- 입력 프롬프트
- 중간 사고 과정 (Chain-of-Thought)
- 도구 호출 내역
- 도구 출력
- 최종 결정

**실패 분류 시스템:**
```
tool_api_timeout    - 도구 API 타임아웃
hallucination_factual - 사실 관계 환각
planning_loop       - 계획 루프
misinterpreted_goal - 목표 오해
```

### 8.3 CI/CD 통합

- 새 에이전트 코드나 프롬프트 배포 전 평가 필수
- 반복 가능하고 확장 가능한 테스트 작업을 파이프라인에 직접 임베드
- 버전 관리 및 자동화

### 8.4 비결정성 처리

**다회 실행:**
- LLM의 확률적 특성으로 동일 입력에 다른 출력 가능
- 단일 실행 테스트는 신뢰할 수 없음
- 여러 번 평가 실행으로 안정적 결과 확보

### 8.5 평가 방법 조합

| 평가 유형 | 용도 |
|----------|------|
| **결정적 규칙** | 형식 준수, 필수 필드 존재 |
| **통계적 검사** | 성능 임계값, 분포 비교 |
| **LLM-as-a-Judge** | 뉘앙스 있는 품질 평가 |
| **인간 검토** | 최종 단계 결정 |

### 8.6 데이터셋 구성

- 도메인 특화, 현실적인 시나리오 개발
- 엣지 케이스 포함 필수
- 정기적으로 새로운 패턴 반영
